PHP Management Layout Options

Goal: manage installed PHP versions at the server level, then handle runtime settings and selected PHP version per site.

Option A: Dedicated PHP Workspace Page
Overview | Sites | Databases | Services | Logs | PHP | Settings
Server level
Installed versions list, install/uninstall, set default CLI, set default for new sites
Site level
Table of sites with current PHP version and quick "Change version" action
Site settings
Per-site limits, OPcache, Composer auth, extensions entry point

Dedicated `PHP` page

Keeps everything PHP-related in one workspace area. Best if PHP should feel like a first-class operations surface beside Services and Sites.

Option B: Server PHP + Site PHP Split
Overview | Sites | Databases | Services | Logs | Settings
Server PHP page/tab
Installed versions
Default CLI + new-site default
Site page
PHP version for this site
Memory/upload/timeout
OPcache and Composer auth

Split by ownership

Server page owns packages and defaults. Each site owns runtime and PHP selection. Best if site configuration should stay close to site deploy settings.

Option C: Settings Tab Expansion
Connection | Keys | Alerts | Inventory | PHP | Notes | Export | Danger
Top card: installed PHP versions and default version controls
Middle card: site mapping table and change actions
Bottom cards: OPcache, extensions, Composer auth, template settings

`Settings` sub-tab

Fastest to fit into the current structure, but risks burying an operationally important area inside the settings section.

Recommendation

I’d lean toward Option B: a server-level PHP management area for installed versions and defaults, with site-specific PHP version and runtime settings on each site. It matches the boundary you described and avoids one large page doing two different jobs.
