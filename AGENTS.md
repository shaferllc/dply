## Learned User Preferences

- Use Tailwind CSS v4 with CSS-first configuration (`@import "tailwindcss"`, `@theme`, and related v4 patterns) for new and updated styling.
- Use one shared site header across marketing and authenticated app layouts; only the navigation should differ for guests versus signed-in users.
- Prefer **[Blade UI Kit Blade Icons](https://blade-ui-kit.com/blade-icons)** for **header and primary navigation** icons, keep the header logo visually prominent, and avoid an overcrowded top bar by moving **overflow links into dropdowns**.
- Use a **shared global footer** (the same **`x-marketing-footer`** component or equivalent) on **app**, **settings**, and **public status** layouts so the footer matches marketing and guest pages and stays visually anchored on short pages.
- Auth screens should pair the form with supporting layout: short value props, icons, or similar context—not a bare form only.
- Aim for an enterprise-ready SaaS look and feel consistent with the product logo and brand colors, and prefer basics-first progressive disclosure so one primary workflow leads while advanced tools are visually secondary; default and auto-fill app-aware setup values where possible, especially on server and site creation flows, and prefer repo-driven setup that auto-detects runtime or framework choices instead of exposing low-level steps unless needed.
- Include a **features** (or equivalent) page that explains **how product capabilities connect**, not only isolated feature lists.
- For BYO, surface **subscription and plan limits** in the product UI and keep them aligned with **docs** (roles, quotas, gates).
- Prefer **Livewire v3 `Form` objects** (or equivalent form components) for forms with **about four or more** fields, instead of many separate `wire:model` properties on the component.
- When using **Livewire `stream()`** for long-running output (for example remote SSH), **only stream during Livewire requests** (`Livewire::isLivewireRequest()`), not on full-page navigations, so full-document HTML is not corrupted.
- Avoid native **`alert()`** and similar browser dialogs for product flows; use **site-styled modals** (or equivalent Livewire/UI patterns) for confirmations and detail content.
- Make notifications universal and easy to attach from anywhere in the product: support quick-add channel flows near the relevant resource UI and prefer reusable modals/components over one-off notification management affordances.

## Learned Workspace Facts

- Multi-product direction is documented in `docs/MULTI_PRODUCT_PLATFORM_PLAN.md`: one shared monorepo and Composer packages, with a **separate database per product** and no shared transactional schema across lines in v1; **example domains in that doc are provisional** until final DNS/product choices. **Default onboarding and operator documentation prioritize the repo-root BYO app**; other `apps/*` products are optional until priorities change (see that plan and `docs/BYO_LOCAL_SETUP.md`). Cross-product auth is moving toward a dedicated **`apps/dply-auth`** app for shared login, 2FA, and redirects back into separate product apps and subdomains.
- Implementation priority for new product lines after BYO: **Serverless**, then **Cloud**, then **WordPress**, then **Edge**. **dply Cloud** is long-running **PHP and Rails** hosting; **dply Edge** is **JavaScript frameworks and static** sites (git, previews, CDN-style delivery), not the default home for Rails monoliths. **dply WordPress** is a **hosted-only managed WordPress** line in v1 on **dply-controlled infrastructure**, not customer-owned VMs or SSH.
- The reusable **serverless engine** now lives in the **root app** under `app/Services/Deploy` and related contracts/support classes, with provider adapters for **AWS Lambda**, **DigitalOcean Functions**, **Cloudflare Workers**, **Netlify**, and **Vercel** plus roadmap stubs. The old standalone serverless product shell has been removed; serverless site creation and deployment stay inside the main BYO app.
- Control-plane schema changes such as introducing **`projects`** and migrating from **`sites`** are scoped to the **BYO app database** until other product apps and databases exist (see ADR-003); reduce **engine leakage** (e.g. SSH-shaped code in non-BYO workers) with **separate queues or worker pools per product**, **adapter-only provider code**, and review discipline.
- BYO **server workspace** uses **separate routes** per sub-area (not primary tab navigation). **Cron, Daemons, Firewall, and SSH keys** in-page sections share **`x-server-workspace-tablist`**, **`x-server-workspace-tab`**, **`x-server-workspace-tab-panel`**, and **`livewire/servers/partials/workspace-ops-not-ready`** for consistent sub-tabs and the shared “Provisioning and SSH must be ready…” empty state. The **provision journey** should keep users in the same page shell, support reconnecting and resuming installation on the same server, surface live and persisted **per-step output/status**, and redirect to the site when setup finishes. Within the server workspace, **Deploy** stays the release surface, **Saved commands** are the server-local runbook area, **Scripts** are org-wide reusable automation, and **Marketplace** imports each item type into one obvious destination. **`Create server`** should stay a **BYO-only** host flow with provider-based provisioning choices, including the richer interactive **plan** and **region** selectors; provider account management belongs on the dedicated providers page, not duplicated in the create flow. Non-BYO deployment models should start from a separate launch or setup chooser that can branch into **Local Docker-backed runtimes** (**Docker** and **Kubernetes**), **Serverless**, **Kubernetes**, **Edge network**, and **Cloud network** paths. Server and site creation should expose **web server selection** with **`nginx`** as the default and clear pros or tradeoff guidance for alternatives. Remote **system logs** and **manage/inventory** actions prefer **root** SSH with **fallback** to the deploy user, driven by **`config/server_system_logs.php`**, **`config/server_manage.php`**, and **`config/server_settings.php`**; **`SshConnection`** supports an optional login username override for that chain.
- Keep **`dply-core`** (or the shared package equivalent) to a **small, stable surface**; expand it deliberately and record boundaries in ADRs when needed. In this monorepo, consume **`shaferllc/dply-core`** via **Composer path** to **`packages/dply-core`** unless a deploy target intentionally uses a **published Git** dependency instead.
- BYO exposes **`/settings`** as a **hub with a shared settings layout** (profile, two-factor, organizations, billing, docs links). **Source control** (`/profile/source-control`) is **Git providers only** (OAuth to GitHub, GitLab, Bitbucket). **Server provider** API tokens for provisioning infrastructure live under the org **`/credentials`** route (settings sidebar: **Server providers**), not mixed with Git. **Per-organization deploy-finish email** can be disabled without affecting **outbound integration webhooks** (separate controls).
- User-facing **Projects** are a grouping container for multiple servers and sites, backed today by **`Workspace`**; they should stay separate from the existing deployment-oriented **`Project`** model, with any naming cleanup treated as optional follow-up rather than a prerequisite.
- **Profile → SSH keys** (`/profile/ssh-keys`) and **HTTP API keys** (`/profile/api-keys`): **user**, **organization**, and **team** **SSH** public keys (where enabled) support optional **provision on new servers** and **deploy** to existing servers via the same **`server_authorized_keys`** sync as server pages. Adding a **personal SSH key** should use one shared modal flow across **server create**, **profile SSH keys**, and **server workspace SSH keys**. **API** tokens are **organization-scoped** with granular abilities from `config/api_token_permissions.php`; optional **`DPLY_API_TOKENS_REQUIRE_PAID_PLAN`** gates **creating** new tokens on **Pro**; **deployer** org members get a reduced ability set when creating tokens.
- **Notification channels** and event routing can be configured at **organization**, **user**, and **team** levels (with **bulk assignment** where the UI supports it). **BYO pricing and limits** are **organization-based**, not seat-based or per-site (**Trial** / **Pro**; limits apply to **all servers and all sites** in an org). **Profile, 2FA, and OAuth-linked accounts** stay **user-scoped**; users should always have an organization, with auto-created defaults **`"<name>'s Workspace"`** and legacy backfill as needed.
- BYO **site settings → DNS** lets operators pick an **organization server-provider credential** for DNS automation (**DigitalOcean** or **Cloudflare** where implemented), an optional **DNS zone (apex)** validated for that account, and uses that choice for **preview/testing hostnames** (falling back to the app **testing-domain** pool) and aligned **DNS-01** / certificate flows. Full arbitrary DNS record editing is not in scope yet; the **Certificates** form still exposes DNS challenge fields when needed.
- Deploying a monorepo app behind nginx: set **`root`** to that app’s **`public/`** (e.g. `apps/dply-cloud/public` in a checkout, or the released app tree’s `public`). **Independent deploys do not require separate Git repositories** if each product has its own env, database, and build or deploy target (see `docs/MONOREPO_AND_APPS.md`). For BYO VM sites, **no-downtime** releases use **`deploy_strategy = atomic`** (release directories + `current` symlink), toggled from site **Settings → Deploy** as **Zero downtime deployment**; **`simple`** updates the live checkout in place.
